# Agent Vs Chat

很多人刚开始学 `Agent` 时，第一反应是：

“它是不是就是更高级的聊天机器人？”

这个理解不完全错，但不够准确。

## 普通 chat 在做什么

普通 chat 的核心任务通常是：

根据你给它的输入和上下文，生成一段回复。

它最擅长的是：

- 解释
- 改写
- 总结
- 对话

但它通常只是“说”，不一定会“做”。

## `Agent` 多了什么

`Agent` 比普通 chat 多出来的关键能力是：

- 判断是否需要行动
- 调用 `Tool`
- 获取工具执行结果
- 根据结果继续处理

也就是说，`Agent` 不只是 reply，它还会 act。

## 一个直观例子

如果你问：

“现在几点了？”

普通 chat 可能会：

- 根据训练时学到的知识胡乱猜一个答案
- 或者直接说它不知道

但一个带 `get_time` 工具的 `Agent` 可以：

1. 看懂你的问题是在问时间
2. 决定调用 `get_time`
3. 读取工具返回的时间
4. 再把这个时间告诉你

这就是为什么 `Agent` 的核心不是“更会说话”，而是“能基于判断采取动作”。

## 暂时先记住一句话

普通 chat 更像：

“我根据上下文生成回复。”

而 `Agent` 更像：

“我先判断要不要做事；如果需要，我去做，再把结果告诉你。”

## `Agent` 一定要有 `LLM` 吗

不一定。

更宽一点地说，只要一个系统具备这些特征：

- 接收输入
- 根据某种策略做决策
- 执行动作
- 观察结果
- 继续回复或继续行动

它就已经有 `Agent` 的基本形态。

这里的“决策策略”不一定非要是 `LLM`，也可以是：

- 规则
- 状态机
- 搜索算法
- 传统程序逻辑

## 为什么现在很多人一提 `Agent` 就默认想到 `LLM`

因为在今天更主流的 `AI Agent` 语境里，人们通常指的是：

- 有 `LLM`
- 能理解目标
- 能决定是否调用 `Tool`
- 调完工具后还能继续处理结果

所以你以后会同时遇到两种说法：

- 广义的 `Agent`
- 狭义的 `LLM-based agent`

这两种说法都存在，只是语境不同。
