# Upgrade Path

这章的目标不是马上把你拖进复杂实现，而是让你知道下一步该怎么走。

## 三层心智模型

理解从规则型 agent 升级到 LLM-based agent，最简单的方法就是把系统拆成三层。

### 1. LLM 层

这一层负责“想”。

它看到的是消息历史、当前用户输入，以及程序提供的工具说明。然后它会判断：

- 现在能不能直接回答
- 需不需要先调用一个工具
- 工具返回结果后，接下来怎么组织回复

你可以把这一层理解成“会参与决策的模型”。

### 2. Tool 层

这一层负责“做”。

工具本身还是普通程序函数，比如：

- `get_time`
- `save_note`
- `read_notes`

工具层的重点不是聪明，而是稳定、清楚、可执行。模型只负责提出请求，真正完成动作的是这些工具。

### 3. Orchestration / Loop 层

这一层负责“串起来”。

它的工作是把整个循环跑通：

1. 收到用户输入
2. 把历史消息和工具信息发给模型
3. 读取模型的决定
4. 如果需要，就执行工具
5. 把 `tool_result` 送回模型
6. 直到模型给出最终回答

这一层是程序最像“管道”的部分。它不替模型思考，也不替工具做事，主要负责调度。

## 这个模型怎么帮你理解升级

如果你把系统分成这三层，就会很容易看明白升级到底发生在哪里：

- 不是把工具删掉
- 不是把程序变成模型
- 而是把“决定下一步”的一部分，交给 LLM 层

这也是从规则型 agent 走向 LLM-based agent 的关键一步。

## 你可以先记住的迁移方向

最自然的升级路线通常是：

1. 保留现有工具
2. 把死板的规则判断换成模型判断
3. 继续让程序负责执行和回传结果

这样做的好处是，变化集中在 `LLM` 和 orchestration 上，工具本身可以先不动。

## 一句话总结

升级不是“加一个大模型就完事了”，而是把系统拆成 `LLM 层`、`Tool 层` 和 `Loop 层`，再让它们各做各的事。
