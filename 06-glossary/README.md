# Glossary

这里不是 API 参考手册。它的作用是帮你在看第 05.5 章或写代码时，快速找回这些术语是什么意思，以及它们为什么重要。

## 核心概念

### Agent

一个不只是会回复，还会决定下一步做什么、调用什么工具、再根据结果继续处理的系统。

### SDK

给开发者用的工具包，通常把底层 API 封装得更好用，让你少写很多重复代码。

### prompt caching

把重复出现的提示内容缓存起来，下次再用时可以少算一部分，通常能节省时间和成本。

### token

模型处理文本时用的基本单位，不等于字数。token 越多，通常意味着上下文和成本也越高。

### max_tokens

一次输出允许的最大 token 数。它会影响回复长度，但不一定等于最终会输出的全部内容。

### thinking

模型在给出答案前的内部推理过程。它能帮助模型把复杂问题想清楚，但不是给用户看的正文。

### effort

可以理解成“愿意花多少力气思考”。effort 越高，模型通常会更认真处理复杂任务，但也可能更慢。

## API 和消息

### Messages API

一种围绕 `messages` 来组织输入和输出的接口方式。你把对话、指令和结果按消息结构传进去，模型再按消息结构返回。

### messages

对话记录的列表。每条消息通常都带有角色和内容，帮助模型知道“谁在说话、在说什么”。

### role

消息的身份标记，比如系统、用户、助手，或工具消息。它决定了这条消息在对话里的作用。

### system prompt

最先给模型的高优先级指令。它常用来定规则、定风格、定边界，比如“用中文回答”或“不要编造”。

### stop_reason

模型为什么停下来。常见情况是回答完成、达到长度上限，或者需要工具先处理下一步。

### Streaming

边生成边返回结果的方式。它适合做“打字机效果”或让前端更快看到内容开始出现。

## 工具和循环

### tool_use

模型发出的“我要用工具”的请求。它通常会说明要调用哪个工具，以及需要哪些参数。

### tool_result

工具执行完后返回给模型的结果。模型会把它接回上下文里，再继续决定下一步。

### agent loop

代理的标准循环：观察输入，决定行动，调用工具，读取结果，再继续处理。很多 Agent 系统都靠它运转。

### manual loop

由开发者或应用自己控制循环的方式。你决定什么时候把结果送回模型，什么时候继续下一轮。

### Tool Runner

SDK 提供的辅助工具，帮你自动跑完整的 tool-calling loop。它不是泛指任何运行时执行器，而是更偏向“帮你把工具调用流程接起来”的便利封装。

### MCP

`Model Context Protocol` 的缩写。一种让模型连接外部工具和数据源的通用协议，重点是统一接入方式。

### approval / human-in-the-loop

在某些敏感操作前先让人确认一下。这样可以减少误删、误发、误操作带来的风险。

### side effect

一次动作对外部世界造成的变化，比如发消息、写文件、扣款。它和“只读查询”不一样，要更谨慎。

### structured output

让模型按固定格式输出，比如 JSON、表格字段或指定字段集合。它更适合后续程序继续处理。

## 运行与调试

### retry

失败后再试一次。它常用于网络错误、临时限流或工具短暂不可用，但不适合无限重试。

### observability / logging

观察系统怎么运行的能力。logging 是最常见的一部分，能帮你看见请求、结果、错误和关键步骤。
