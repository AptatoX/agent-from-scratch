# 从 Tool Calling 到 LLM Agent

如果你已经知道“大模型能聊天”，但还不太明白“它为什么突然会查资料、调工具、像助手一样连续做事”，这篇文章适合你读。

这篇文章不是 API 手册，也不是某一家厂商的功能清单。它更像一张给新手用的地图：先把整体结构看明白，再去读具体 SDK、参数和代码，你会轻松很多。

可选阅读：如果你想先补前置概念，可以先看 `05.5-from-rules-to-llm-agent` 和 `06-glossary`，再回来看这篇长文。这样会更容易把这里的术语放回整体语境里。

---

## 新手先建立一个总感觉

先记一句最重要的话：

**Agent 不是“会聊天的模型升级版”，而是“模型负责判断，程序负责执行，二者通过循环协作”的系统。**

很多新手第一次接触 agent，会把注意力放在很碎的术语上，比如 `tool_use`、`schema`、`max_tokens`、`MCP`。这些词当然重要，但如果一开始就一个个背，通常会越学越乱。

更好的顺序是先抓整体：

1. 先理解模型本身只负责“看、想、决定”
2. 再理解真正动手做事的是外部工具
3. 最后理解 agent 其实就是把“思考”和“执行”串起来的循环

只要这个骨架先立起来，后面的术语基本都能自然落位。

---

## 什么是 Agent

先从最容易混淆的地方讲。

平常我们说“调一次大模型”，通常指的是一轮问答：

- 你问一个问题
- 模型给一个回答
- 这一轮结束

而 **Agent** 更像是一个围绕目标持续行动的系统。你给它的不只是一个问题，还可能是一个任务，一个目标，或者一个模糊的需求。它需要判断：

- 现在先做什么
- 需不需要用工具
- 工具结果拿到以后下一步做什么
- 什么时候可以停下来给出结果

所以可以把 Agent 理解成：

> 一个不只会生成回答，还会决定下一步行动，并根据外部结果继续推进任务的系统。

一个常见类比是：把模型想成一个“会做计划的实习生”。

- 它会分析任务
- 它会说自己需要什么信息
- 它会提出“请帮我查一下”“请帮我执行一下”
- 但它自己并不能直接改数据库、发邮件、读你电脑里的文件

真正去做这些事的，是你提供给它的工具，以及承接这些工具的程序。

---

## 什么是 Tool / Tool Use

### 什么是 Tool

**Tool** 就是给模型使用的外部能力。

比如：

- 搜索资料
- 读文件
- 调天气接口
- 查数据库
- 发消息
- 执行命令

模型本身擅长理解语言、做推理、组织输出，但它不是天然就拥有这些外部执行能力。它不能因为“知道天气是什么”，就真的去实时查今天北京有没有下雨。它也不能因为“懂数据库概念”，就自动连上你的生产库。

所以工具的本质是：

> 把外部世界中的某种可执行能力，包装成模型可以请求使用的接口。

### 什么是 Tool Use

**Tool Use** 指的是整套“模型请求工具，程序执行工具，再把结果送回模型”的过程。

它不是单指某个工具，也不是单指某次调用动作，而是一种配合机制。

可以先记成这四步：

1. 你把可用工具告诉模型
2. 模型判断现在要不要用工具
3. 你的程序真正执行工具
4. 执行结果再回给模型

这件事为什么重要？因为它决定了模型从“只会说”变成“可以借助外部能力做事”。

---

## 工具调用到底是怎么发生的

很多人第一次看到 Tool Calling，会误以为是“模型自己去执行工具”。其实不是。

真正发生的过程更接近下面这样：

1. 用户给出问题或任务
2. 你的程序把这次对话内容和工具定义一起发给模型
3. 模型阅读这些信息后做判断
4. 如果它认为不需要工具，就直接回答
5. 如果它认为需要工具，就返回一个结构化的工具调用请求
6. 你的程序读取这个请求，执行对应工具
7. 你的程序把执行结果再发回模型
8. 模型基于新结果继续回答，或者继续请求下一个工具

这里最容易记错的一点是：

**模型负责“提出调用请求”，不是负责“亲手执行调用”。**

所以职责分工非常清楚：

- 模型负责决策
- 程序负责执行
- 工具负责接触外部世界

如果你把它想成团队协作，会更容易理解：

- 模型像项目负责人，知道现在该做什么
- 工具像不同岗位的同事，分别负责查资料、发请求、读文件
- 你的程序像调度系统，负责把话传对、把结果接回

---

## 从 API 角度看，这个循环长什么样

从 API 视角看，工具调用不是一锤子买卖，而是一个循环。

最简化的流程是：

1. 客户端向模型 API 发起请求
2. 请求里带上：
   - 当前用户消息
   - 历史消息
   - system prompt
   - 可用工具定义
3. 模型返回内容
4. 你检查这次返回是“直接结束”还是“要求调用工具”
5. 如果要求调用工具，就执行工具
6. 把工具结果追加到消息历史
7. 再次调用模型 API
8. 一直循环到模型真正结束

如果只压缩成一句话，就是：

**用户消息 -> 模型判断 -> 工具执行 -> 结果回传 -> 模型继续**

这就是后面常说的 `agent loop` 的雏形。

## 什么是 Messages API

很多现代大模型接口都以“消息列表”的形式组织上下文。无论具体名字是否相同，你都可以把 **Messages API** 理解成：

> 用一串有角色的消息，把这轮对话和所需上下文交给模型处理的接口方式。

它通常会包含这些信息：

- `messages`：对话历史
- `system` 或类似字段：系统级指令
- `tools`：模型可用的工具定义
- `max_tokens`：输出长度上限
- 以及模型名、流式设置等其他选项

为什么它重要？因为 Tool Calling 不是脱离对话存在的。模型之所以知道“现在该不该调工具”，依赖的就是你提供给它的这一整包上下文。

---

## 什么是 messages / role / system prompt

这三个词通常一起出现，最好一起理解。

### 什么是 messages

`messages` 就是对话历史，也可以理解成“这次请求带给模型看的上下文记录”。

它告诉模型：

- 用户刚刚说了什么
- 之前说过什么
- 模型自己之前回复过什么
- 工具结果是什么

一个很重要的初学者认知是：

**API 通常是无状态的。**

也就是说，模型不会神奇地“自动记住上一次接口调用发生了什么”。如果你想让它知道历史，就必须把历史继续传进去。

### 什么是 role

`role` 表示这条消息是谁发出的。

常见角色包括：

- `user`
- `assistant`
- `system`

有的接口里，工具结果会放在带特殊结构的内容块里；有的接口会把它们包装在用户消息里；有的还会区分更细的角色。具体写法可能不同，但核心思想一致：

> 让模型知道“这段内容是谁说的，它在对话里扮演什么作用”。

### 什么是 system prompt

`system prompt` 是你给模型的高优先级说明书。

它通常用来规定：

- 你是什么角色
- 你应该怎么回答
- 有哪些边界不能越过
- 看到某类情况时优先采取什么策略

对新手来说，可以把它理解成“后台规则”，而 `user message` 更像“前台需求”。

---

## 什么是 schema

**Schema** 可以先理解成“参数格式说明书”。

比如你有一个天气工具，它可能要求输入里必须有一个 `city` 字段，而且这个字段必须是字符串。这个规则本身，就是 schema 要表达的内容。

它的作用至少有三层：

1. 告诉模型这个工具要怎么用
2. 限制模型不要乱传参数
3. 帮你的程序做输入校验

schema 写得越清楚，模型调用工具通常越稳定。因为模型不是在读你脑子里的设想，它只会根据你提供的描述和格式约束来判断。

对初学者最值得先记住的一句经验是：

**工具描述和 schema 越含糊，工具调用越容易漂。**

---

## 什么是 stop_reason / tool_use / tool_result

这三个词连在一起看，最容易明白。

### 什么是 stop_reason

`stop_reason` 表示：模型这一次为什么停下来了。

最常见的几类意思是：

- 它已经回答完了
- 它输出达到上限了
- 它现在想调用工具
- 它处于某种可以继续的暂停状态

对 Tool Calling 来说，最关键的是分清这两种：

- “已经可以交答案了”
- “先别交答案，我要你帮我执行工具”

### 什么是 tool_use

`tool_use` 可以理解成模型开出的一张“工具调用单”。

它通常会包含：

- 要调用哪个工具
- 这次调用的参数
- 一个唯一标识，方便后续把结果对应回去

也就是说，模型并不是返回一句自然语言说“请帮我查天气”，而是返回一个可供程序处理的结构化请求。

### 什么是 tool_result

`tool_result` 就是你的程序执行完工具之后，发回给模型的结果。

它的作用是告诉模型：

> 你刚才请求的那个动作，我已经做完了，这是结果，请你基于这个结果继续判断。

如果工具失败了，也不一定非要直接崩掉。很多实现会把错误也包装成一种结构化结果回传给模型，让模型决定是换一种办法、追问用户，还是直接解释失败原因。

---

## 什么是 agent loop / manual loop / Tool Runner

### 什么是 agent loop

`agent loop` 就是 agent 的主循环。

它反复做这几件事：

1. 读取当前上下文
2. 调模型
3. 看模型是结束还是调工具
4. 如果要调工具，就执行工具
5. 把新结果塞回上下文
6. 再调下一轮

之所以叫 loop，不是为了显得高级，而是因为它真的会一轮一轮跑。

### 什么是 manual loop

`manual loop` 指的是这整个循环由你自己手写和控制。

好处是很灵活。你可以自由加入：

- 审批
- 权限控制
- 重试
- 超时
- 日志
- 自定义停止条件

代价是你要自己处理更多细节。

### 什么是 Tool Runner

`Tool Runner` 则是更高层的封装。它的思路是：

> 这套循环太常见了，SDK 先帮你包一层。

所以你不一定每次都要从零手写 loop。有些 SDK 会直接提供“自动处理工具调用闭环”的能力，适合快速起步。

它并不意味着“完全不用理解 loop”，而是意味着“你理解 loop 之后，可以选择不自己写底层样板代码”。

## 自己开发 agent 时，可以先记住哪三层

如果你准备自己实现一个 agent，先不要急着把所有名词都抄进笔记里。先记住三层，就已经抓住主干了。

### 第一层：LLM 层

这一层负责：

- 理解目标
- 读取上下文
- 决定要不要调用工具
- 基于结果继续推理

它负责“判断”。

### 第二层：工具层

这一层负责：

- 连接外部能力
- 真正执行动作
- 把现实世界中的信息或动作暴露给模型

它负责“执行”。

### 第三层：调度层

这一层负责：

- 维护消息历史
- 调模型 API
- 解析工具请求
- 执行工具
- 回填工具结果
- 控制循环结束

它负责把前两层串起来。

对新手来说，只要把这三层分开，很多问题都会清晰很多。比如：

- “这是模型能力问题，还是工具能力问题？”
- “这是 schema 没写清楚，还是 loop 没处理好？”
- “这是要改 system prompt，还是要改工具权限？”

---

## 什么是 SDK / Streaming / token / max_tokens

到这里，你已经知道 agent 的主结构了。接下来可以往前再走一步：当你真的准备把它写出来、跑起来时，最先遇到的往往不是更复杂的 agent 概念，而是一些很基础、但会反复碰到的运行问题。

比如：

- 你用什么方式去调 API
- 返回内容是一次性拿到，还是边生成边返回
- 为什么同样一段话，有时更贵，有时更慢
- 为什么回答会被截断

`SDK`、`Streaming`、`token`、`max_tokens`，就是回答这些问题的第一组概念。

### 什么是 SDK

**SDK** 是官方或社区提供的开发工具包，用来帮你更方便地调用模型 API。

它通常会帮你处理：

- HTTP 请求细节
- 认证
- 参数组织
- 响应解析
- 流式输出
- 错误类型
- 有时还包括 Tool Runner 之类的高级封装

所以 SDK 的价值不是“让原理变化”，而是“让实现更省力”。

### 什么是 Streaming

**Streaming** 就是流式返回。

不流式时，模型通常是“全部生成完，再一次性返回”。

流式时，模型是“边生成边返回”，所以用户可以更早看到内容开始出现。它特别适合：

- 长回答
- 等待时间敏感的交互
- 需要展示过程感的界面

### 什么是 token

**token** 是模型处理文本时的基本计量单位。

它不等于字数，也不等于单词数，但你可以先把它当成“模型看文本和产出文本时使用的长度单位”。上下文越长、输出越长，token 通常就越多，成本和延迟也往往会跟着增加。

### 什么是 max_tokens

`max_tokens` 是这次请求允许模型最多输出多少 token。

它更像“输出长度上限”，不是保证一定输出那么多，而是告诉模型“最多到这里为止”。如果设得太小，回答可能被截断。

## 什么是 thinking / effort / prompt caching

当你把“怎么调 API、怎么接收输出、长度怎么算”这些最基础的问题搞清楚以后，下一层自然会碰到的是：

- 同一个任务，模型为什么有时答得更浅，有时更认真
- 为什么有些请求更慢，但结果可能更稳
- 为什么一段很长的固定前缀会影响成本

这时候，讨论重点就从“接口怎么调”转向“模型这次要以什么强度工作，以及上下文怎么更高效地复用”。这就是 `thinking`、`effort`、`prompt caching` 这一组概念的意义。

### 什么是 thinking

`thinking` 可以理解成更明确地启用或配置模型的推理过程。

对新手来说，不需要先把实现细节抠得很深，只要先知道：

- 简单任务不一定需要很强的推理配置
- 复杂任务通常更依赖推理能力
- 推理更强，通常也意味着更高延迟或更高成本

### 什么是 effort

`effort` 可以理解成“愿意投入多少思考强度”。

一般来说：

- 低 effort 更快
- 高 effort 可能更稳
- 但更高 effort 不代表一定更好，它只是适合更复杂、更值得花力气的问题

### 什么是 prompt caching

`prompt caching` 是一种成本和延迟优化机制。

如果很多请求共享同样的前缀内容，比如：

- 很长的 system prompt
- 很长的固定背景资料
- 一组基本不变的工具定义

那么这些重复部分就有机会被缓存，从而减少重复计算。

你可以把它理解成：

> 不是每次都从头重算整包上下文，而是尽量复用那些反复出现的部分。

## 什么是 MCP / approval / side effect / structured output / retry / observability

如果说上一组概念讨论的是“模型这一轮怎么工作得更合适”，那再往下一步，就是“这个系统进入真实环境以后，怎样才能既可用，又可控”。

这时你考虑的问题会继续变化：

- 工具是不是能用更统一的方式接入
- 哪些动作不能让模型直接自动执行
- 输出给下游程序时要不要固定格式
- 失败后应不应该重试
- 系统出了问题时你能不能看见它卡在哪里

所以这一部分不再只是模型参数问题，而是 agent 工程化时绕不过去的配套能力。

### 什么是 MCP

**MCP** 可以先粗略理解成一种标准化接入外部工具和上下文资源的协议。

它想解决的问题是：不同工具、不同服务、不同数据源，能不能用更统一的方式接给模型或 agent 系统。

对新手来说，先不用把它神化。你只要先知道：

> 它是在“模型如何接外部能力”这件事上做标准化。

### 什么是 approval

`approval` 指的是审批或人工确认。

也就是说，模型提出“我要执行这个动作”以后，不是立刻自动执行，而是先过一层人类确认。

这在涉及副作用操作时尤其重要。

### 什么是 side effect

`side effect` 指会真实改变外部世界状态的动作。

比如：

- 发邮件
- 删文件
- 写数据库
- 发起支付
- 创建工单

和它相对的是只读操作，比如查询天气、读日志、搜文档。副作用越强，通常越应该加审批、权限和审计。

### 什么是 structured output

`structured output` 指按指定结构输出结果，而不是自由散文式回答。

比如让模型输出 JSON、固定字段、标准表格行。这样做的价值很直接：程序更容易接，后续流程更稳定。

### 什么是 retry

`retry` 就是重试。

但它不是“报错了就无脑再来一次”。真正合理的重试，通常会区分：

- 哪些错误是临时的，适合重试
- 哪些错误是参数错、权限错，不该重试
- 重试几次
- 每次间隔多久

### 什么是 observability

`observability` 可以理解成系统的可观察性。

你是否能看见：

- 每轮发了什么请求
- 模型为什么停下
- 调了哪些工具
- 工具耗时多少
- 哪一步经常失败
- token 花在哪儿

如果这些都看不见，agent 一旦出问题，你基本只能靠猜。

## 新手该先建立什么整体理解

现在可以回头看一眼：前面后半段其实讲了三件连续发生的事。

第一件事，是把 agent 跑起来，所以你会遇到 SDK、Streaming、token、max_tokens。

第二件事，是让模型在一次请求里工作得更合适，所以你会遇到 thinking、effort、prompt caching。

第三件事，是把 agent 放进真实环境里，所以你会遇到 MCP、approval、side effect、structured output、retry、observability。

顺着这条线再回头看，整篇文章就不是一串分散术语，而是一条从“能跑”到“更稳”再到“可落地”的学习路径。

### 第一层理解：模型不是执行器

模型最核心的能力是理解、判断、推理、生成，不是直接操作外部世界。

### 第二层理解：工具是执行接口

工具让模型有机会借用外部能力，但真正执行动作的仍然是你的程序和工具实现。

### 第三层理解：agent 是循环，不是一次调用

只调一次模型，通常只是问答。

能在“判断 -> 调工具 -> 拿结果 -> 继续判断”里跑起来，才更接近 agent。

### 第四层理解：工程问题和概念问题同样重要

当你从 demo 走向真实项目时，很多真正决定系统能不能用的点，不是“提示词再加一句”，而是：

- schema 清不清楚
- 有没有审批
- 副作用是否受控
- 重试是否合理
- 有没有日志和可观察性
- 输出能不能结构化接入

### 第五层理解：先会搭骨架，再追求高级能力

对新手最稳的学习顺序通常是：

1. 先理解普通消息调用
2. 再理解工具调用
3. 再理解 loop
4. 最后再碰更复杂的 agent runtime、MCP、多工具协同、长期记忆之类的话题

这样不容易把自己淹没在名词里。

---

## 最后一句总结

从 Tool Calling 到 LLM Agent，本质上不是“模型突然学会了做事”，而是“模型学会了提出行动请求，而你的程序把这些请求组织成一个能持续运行的协作循环”。
